# Apex cutover requirements matrix

Last reviewed: 2026-07-22. This matrix is the acceptance ledger for the Alpha
OS V1 to Apex OS V2 transition. It is derived from the supplied NCV scheduling
requirements, product roadmap, 2026-07-21 meeting transcript, and employee
workbook. It does not treat a code path as proof that a vendor account, roster
fact, migration, or production workflow has been accepted.

## Status definitions

- **VERIFIED**: executable repository evidence exists and passes on the exact
  candidate commit. A live dependency is not implied unless the evidence column
  explicitly names a live test.
- **PREPARED / EXTERNAL**: Apex contains a safe boundary or implementation, but
  named input, credentials, data, rehearsal, or owner acceptance is still
  required. This is a go-live blocker when the requirement is enabled at launch.
- **BLOCKED**: Apex cannot safely finish or enable the requirement with the
  evidence currently available.
- **OUT OF SCOPE**: the meeting assigned or deferred the work outside this Apex
  cutover. It must not silently become a go-live dependency.

## Go-live interpretation

`npm run cutover:preflight` proves static safety and implementation evidence.
`npm run cutover:go-no-go` additionally requires every external approval flag and
must remain **NO-GO** until the underlying artifacts exist. Environment flags
record a reviewed change decision; they are never substitutes for the roster,
credential, vendor, reconciliation, pilot, security, QA, and rollback evidence
listed above.
